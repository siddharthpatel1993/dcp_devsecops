DevSecOps CI/CD Pipeline
AI Learning Platform

Production-Grade | 18-Stage Pipeline | Canary Deployments | Observability-Driven Rollbacks

Problem Statement

Needed an end-to-end DevSecOps pipeline for a Django-based AI learning application integrating security at every stage — from commit to canary production deployment on Kubernetes with automated observability-driven rollbacks.

Application

  • Django web app using OpenAI API + YouTube Transcript API to generate learning content
  • Served via Gunicorn (3 workers) on port 8000
  • Multi-stage Dockerfile (900MB → 150MB) with .dockerignore
  • AI Chatbot tutor powered by GPT-3.5-turbo

Architecture (Production Ready)

Developer → Git Push → Jenkins Pipeline (18 Stages)
    │
    ├── Secret Scan (Trivy) → SCA (Snyk) → Unit Tests (pytest)
    ├── SAST (SonarQube) → Quality Gate
    ├── Docker Build (multi-stage) → Trivy Image Scan (gate)
    ├── Push → Cosign Sign → S3 Reports
    ├── Deploy to DEV (auto) → DAST (OWASP ZAP) → Smoke Test
    ├── Promote to STAGING (auto) → Smoke Test
    ├── Manual Approval → Deploy to PROD (Canary: 5%→20%→50%→80%→100%)
    └── Prometheus Analysis → Auto-Rollback on failure

CI/CD Platform Infrastructure (Built from Scratch)

  • Docker Compose stack: Jenkins Master + Custom Agent (Sonar Scanner + Trivy) + SonarQube
  • Custom Jenkins Agent Dockerfile: Ubuntu + SSH + Sonar Scanner 5.0.1 + Trivy + Java JDK
  • install.bash — bare-metal alternative
  • Result: Entire DevSecOps environment reproducible in <5 minutes

Security Layers (Defence in Depth)

Layer 1 — Secrets

Pre-commit hooks (detect-secrets) + GitHub Push Protection + Trivy CI scan (hard gate)

Layer 2 — Dependencies (SCA)

Snyk scan (requirements.txt + transitive deps) + License compliance

Layer 3 — Code (SAST)

SonarQube SAST + pytest coverage + Quality Gate enforcement

Layer 4 — Container

Trivy scan (gate before push), multi-stage Dockerfile, .dockerignore

Layer 5 — Supply Chain

Cosign signing, Git-SHA tags, Kyverno policy (reject unsigned at admission)

Layer 6 — Runtime

OWASP ZAP DAST, Canary analysis (auto-rollback), K8s probes, resource limits, Prometheus alerts

Kubernetes Deployment (Production Ready)

  • Argo Rollout: 3 replicas + canary strategy (5%→20%→50%→80%→100%) with Prometheus analysis
  • Probes: readiness + liveness (httpGet, auto-restart on deadlock)
  • Resources: requests + limits (prevents noisy neighbor, enables HPA)
  • Service: ClusterIP (stable) + ClusterIP (canary) for traffic splitting
  • Ingress: Nginx + TLS via cert-manager (Let's Encrypt auto-renew) + canary annotations
  • PDB: minAvailable: 2 (safe node maintenance)
  • Kyverno: Reject unsigned images + restrict registries
  • Multi-Environment: Kustomize overlays (dev/staging/prod) — same image, different config

Canary Deployment with Auto-Rollback

5% traffic → Prometheus check → PASS → 20% → PASS → 50% → PASS → 80% → PASS → 100%
                                 FAIL → AUTOMATIC ROLLBACK (0% canary, stable serves all)

Metrics checked at each step:
  ✓ Success Rate > 95% (HTTP non-5xx)
  ✓ P99 Latency < 500ms
  ✓ Error Rate ≤ 2x stable version

Result: 95% of users NEVER see the bug. Rollback within 2 minutes.

GitOps with ArgoCD

  • Jenkins updates image tag in GitOps repo → ArgoCD detects → canary rollout
  • Jenkins NEVER touches cluster (no SSH, no kubectl)
  • Drift detection: manual kubectl changes auto-reverted (selfHeal: true)
  • Rollback = git revert → ArgoCD syncs previous state
  • Audit: git log = who deployed what, when
  • 3 ArgoCD Applications: DEV (auto-sync) | STAGING (auto-sync) | PROD (manual sync)
  • Kustomize overlays: base/ + overlays/dev + overlays/staging + overlays/prod

Observability Stack

Prometheus

  • ServiceMonitor — auto-discovers app + ingress metrics
  • AnalysisTemplate — Prometheus queries as canary gates
  • Alert Rules: HighErrorRate, HighLatency, PodRestarts, CanaryRollback

Grafana Dashboard

  • Traffic split (canary vs stable req/s)
  • Error rate comparison (canary vs stable %)
  • Latency P50/P99 comparison
  • HTTP status codes, pod restarts, replica count

Key Design Decisions

DecisionRationale
Multi-stage Dockerfile900MB → 150MB, no build tools in prod
Git-SHA tagsImmutable, traceable — never :latest
Secret scan firstFail fast — catch credentials before wasting build time
SCA before Docker buildDon't waste compute building image with vulnerable deps
Scan BEFORE pushVulnerable images never reach registry
Cosign + KyvernoOnly CI/CD-built images run in cluster
Multi-env promotionSame image through dev→staging→prod — "what you test = what you deploy"
Canary over Rolling Update95% users unaffected during bad deploy vs 100% affected
Prometheus-driven rollbackMachine-speed detection — rollback in 2 min, not 30 min (human)
ArgoCD manual sync (prod)Double safety — Jenkins approval + ArgoCD approval
ClusterIP + IngressHTTPS, domain routing — not NodePort
3 replicas + PDBSurvives failures, safe maintenance
DAST as promotion gateCatches runtime issues SAST can't see — before staging/prod
